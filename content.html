OOB(Object Oriented Programming)

The DL of OOB

  • OOB is a programming paradigm and not a programming language. Programming languages can be OOB programming languages
  • OOB programming is commonly used in many projects large and small because of the modularity, dynamism, and reusability of code written in a OOB manner
  • When programming in an OOB you will be able to reuse function(methods) that are defined in a class so you can reuse blocks of code to repeat the same thing but for different objects
  • It is advantageous to program with an OOB model so you don't have to repeat things similarly to how you don't have to repeat CSS by having it in a separate file that your HTML can reference as many time as you need that styling.

OOB Dictionary

  • Object: An object is a model of what can be done within a class. The object can be a variable, function, and/or data.
  • Class: A Class is expandable code for creating objects
  • Instance: An instance is a specific implementation of a class. In the same way that you have a fish being a class, salmon is a class, and a salmon named mary is an instance.
  • Method: A Method is a procedure/function that is inside of a class
  • Module: A Module is a file that you import to your program to then access it's contents for methods and classes.
  • Library: A library is a repository of prebuilt classes and functions that will help you with tasks that are oriented around the library.
  • Class Variable: A class variable is a variable defined in side of a class that many different instances of the class can take advantage of.
  • Variable Inheritance: Variable inheritance is where a variable is defined in a parent class but is being used by a child class inheriting it. That variable doesn't then need to be defined in the same manner as it is in the parent class
  • Method Overriding: This is a feature that allows a child class to have its method be method be utilized instead of parent method that has already been written.

Servers

  • When a server receives a request it refers to its handlers to determine what it needs to do next.
  • Get requests: when a server receives a get request it is returning content requested from a specific source
  • Post requests: when a server receives a post request it is when data is submitted to be processed in som e way.
  • Get requests remain in browser history because they are part of the URL whereas Post requests are not sa ved
  • Get requests not be used for sending passwords and other information that should be safe. A Post request should be used for this
  • There are limitations to Get requests you can send up to 64k through the request line. Post requests have few or no restrictions on what can be sent
